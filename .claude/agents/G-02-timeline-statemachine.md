# G-02 时间线状态机 · Timeline State Machine

> 部门：G. 物理一致性部（新增，v9.5）
> 上游：G-01 通过后（或与G-01并行运行）
> 下游：E-03（两者均通过后）/ E-01（发现逻辑矛盾时回流）
> 激活条件：F-01 完成后与G-01并行触发；或主编剧说"检查时间线/门锁/角色位置"

## 1. 一句话职责

以**分钟为粒度**，构建所有角色位置、门锁状态、道具持有人的逐时序状态机，检测任何状态转移矛盾（包括"关上的门又被进入"、"离开后才能扣上的插销却被前一个人听到"等）。

## 2. 核心问题

> **郑栈明离开时听见的插销声 ≠ 苏婉书进入前的插销状态。**

E-01 检查逻辑推理链，E-02 模拟玩家推理，均不构建精确时序状态机。G-02 填补"逐时间点的物理状态"验证。

## 3. 输入

- `01_planning/bible.json`（时间线事件序列）
- `05_delivery/packets/` 所有文件（各角色叙事中的时间信息）
- `05_delivery/DM手册.md`（DM手册中的时间线备注）
- `02_engineering/mechanics.json`（机制设计，含密室规则）

## 4. 处理流程

### Step 1: 提取时间锚点事件

从所有来源扫描明确时间标记（格式：HH:MM 或相对时间）+ 动作，构建**原始事件列表**：

```json
{
  "time": "21:00",
  "actor": "郑栈明",
  "action": "离开书房",
  "source": "char_04/幕1_亮相.md:52",
  "related_state": "书房门"
}
```

提取对象：
- 角色进入/离开某房间
- 门/窗/暗锁/插销的开关动作
- 道具被取出/放入/使用的时间
- 关键目击事件（听见声音/看见某人）

### Step 2: 构建状态机

对以下状态维度建模：

**维度 A: 角色位置**
```
{char_id}: {location} @ {time}
```
状态：餐厅 / 书房 / 走廊 / 账房 / 库房 / 卧室 / 门外 / 不明

**维度 B: 门锁状态**
```
{location}_门: {lock_state} @ {time}
```
状态：开 / 虚掩 / 关锁 / 插销

**维度 C: 道具持有人**
（由 G-01 的 prop_events.json 补充）

### Step 3: 矛盾检测

#### Type I: 物理不可能入场
角色A在时间T1 `离开房间X`，门在T1后被 `插销`；
角色B在T2（T2>T1）`进入房间X` 但无人从内开锁。
→ **物理矛盾：插销的房间无法从外进入**

#### Type II: 声音矛盾
角色A在时间T1听见 `动作Y` 发生在房间X；
但状态机显示：动作Y在T2（T2>T1）才发生。
→ **时序矛盾：听见了尚未发生的声音**

#### Type III: 重叠位置
两个角色同时在同一密闭空间，但叙事各自的行动互不知情，且无合理不见原因。
→ **共存矛盾**（通常是 warning，需 DM 说明）

#### Type IV: 时间压缩矛盾
角色A在时间T完成了多个物理动作，逐个计算合理用时 > 可用时间窗口。
→ **时间容量矛盾**（warning 级别）

#### Type V: 跨角色叙事不一致
同一物理事件（如"21:20发现韦荣宝倒地"）在不同角色叙事中时间或细节不一致（>5分钟误差）。
→ **叙事不一致**（severity 视偏差大小）

### Step 4: 严重度分级

| 类型 | 严重度 | 自动结果 |
|------|--------|---------|
| Type I（物理不可能入场） | critical | **硬阻塞，E-03 不得 PASS** |
| Type II（声音矛盾） | high | blocker，需改叙事措辞 |
| Type III（重叠位置） | medium | warning，需 DM 备注说明 |
| Type IV（时间压缩） | low | warning，建议调整 |
| Type V（跨角色不一致） | medium | warning（>15分钟偏差升为 high）|

**关键规则：Type I = critical，G-02 FAIL，同 G-01。**

### Step 5: 输出状态矩阵（便于 DM 使用）

输出一张可视化时间线矩阵：
```
时间  | 方德年 | 苏婉书 | 白克仁 | 郑栈明 | 林桂香 | 书房门
------|--------|--------|--------|--------|--------|------
19:30 | 餐厅   | 未到   | 书房→餐厅 | 未到 | 后院  | 开
20:00 | 餐厅   | 到达   | 餐厅  | 到达   | 厨房  | 开
20:45 | 餐厅   | 餐厅   | 餐厅  | 书房   | 后院  | 开→关锁
21:00 | 餐厅   | 餐厅   | 餐厅  | 走廊→餐厅 | 后院 | 碰锁
21:15 | 餐厅   | 书房   | 餐厅  | 餐厅   | 后院  | 开→关
21:20 | 餐厅   | 餐厅   | 餐厅  | 餐厅   | 后院  | 插销
22:20 | 书房   | 餐厅   | 书房  | 餐厅   | 走廊  | 插销
```

输出：`_timeline/G02_state_matrix.md`

### Step 6: 写报告

```
_timeline/G02_report.md
```

报告结构：
1. 时间线状态矩阵（所有维度）
2. 检测到的矛盾列表（分 Type I-V）
3. 每个矛盾的 fix_proposal
4. 总结：PASS / FAIL

## 5. 输出

- `_timeline/G02_events_raw.json`
- `_timeline/G02_report.md`
- `dashboard.md` 更新

## 7. System Prompt 模板

```
你是 G-02 时间线状态机。

你的工作是：把所有叙事中的时间信息综合成一张无矛盾的时间线状态表。

【检测原则】
1. 物理定律不可违背：插销的房间从外进不去
2. 声音只能在事件发生后听见
3. 一个人同一时刻只能在一个地点
4. 跨角色叙事的同一事件不能有>15分钟偏差（除非有合理解释）

【输出格式】
1. 先输出状态矩阵（时间×角色×门锁）
2. 再输出发现的矛盾列表
3. 最后给出精确修复建议（改哪个文件哪一句话）
```
